# WillChain Demo Webpage Specification

본 문서는 **WillChain.pdf 제안서의 14페이지 ‘분산원장 작성 시나리오'**를 사용자가 직관적으로 이해할 수 있도록 시연하는 **Demo 웹페이지의 상세 명세서**이다. 이 문서는 단순한 화면 설명이 아니라, **시나리오 데이터 구조**, **스테이지별 상태 변화**, **각 참여 주체의 행동**, **사용자 UI 문구**, **온체인 이벤트 블록 구조**, **승인 절차**, **애니메이션/인터랙션 규칙**, **데모용 가정**까지 포함하는 구현 기준 문서이다.

이 문서를 읽기 전에 반드시 **WillChain.pdf 제안서 전체**, 특히 다음 내용을 먼저 이해해야 한다.

- 블록체인 네트워크 참여 기관 구조
- 유언 등록 / 수정 / 사망 확인 / 사후 절차 개시 / 유언 열람 승인 / 보험금 지급 / 상속·증여 집행 / 재산 토큰화 이벤트 정의
- 허가형 블록체인(permissioned blockchain) 구조
- 온체인에는 원문이 아니라 **이벤트 로그와 해시/참조정보**만 기록하고, 실제 문서는 오프체인에 보관하는 구조
- 보험사가 오퍼레이터이자 오라클 역할을 수행하는 네트워크 구조
- 14페이지의 시나리오 순서

본 데모 웹페이지는 **React + Typescript + Tailwind CSS**를 기반으로 구현하며, 전체 톤은 다음을 지향한다.

- 심플하고 세련된 디자인
- 금융/공공 서비스 같은 신뢰감 있는 분위기
- 발표/시연에 적합한 명확한 정보 구조
- 블록체인 개념을 모르는 사람도 흐름을 이해할 수 있는 UI
- 복잡한 법률/행정 절차를 “한눈에 보이는 단계형 인터랙션”으로 풀어낸 경험

# 1. 문서의 목적과 구현 원칙

## 1.1 목적

이 데모는 실제 상용 서비스가 아니라, **WillChain 개념 증명 및 시나리오 시연용 웹페이지**이다. 따라서 목적은 다음과 같다.

1. **현실 세계에서 어떤 일이 일어나는지**를 보여준다.
2. 그 현실 사건이 **어떤 온체인 이벤트 블록으로 기록되는지**를 연결해서 보여준다.
3. 각 이벤트마다 **누가 승인하는지**, **왜 승인하는지**, **승인 후 어떤 상태가 되는지**를 보여준다.
4. 유언/자산/스마트 컨트랙트/실물자산토큰/계좌 상태가 단계별로 어떻게 바뀌는지 보여준다.
5. 사용자가 우측 UI를 통해 “알림 확인”, “승인”, “자동 열람 결과 확인” 등을 체험하게 해, 블록체인 상속·증여 시스템의 동작 원리를 이해시키는 것을 목표로 한다.

## 1.2 중요한 구현 원칙

이 웹페이지는 실제 블록체인 노드, 실제 전자서명, 실제 금융 API와 연동하지 않는다. 대신 다음 원칙을 따른다.

- **현실 절차와 온체인 기록을 분리해 시각화한다.**
- **모든 이벤트는 Stage 단위로 재생 가능해야 한다.**
- **어느 시점이든 특정 stage로 점프 가능해야 한다.**
- **같은 데이터 구조를 기반으로 좌측 참여자 상태 / 중앙 자산 상태 / 우측 사용자 UI / 상단 시나리오 바 / 블록 hover 정보를 모두 렌더링할 수 있어야 한다.**
- 구현자는 stage 데이터만 수정하면 전체 데모 내용을 조정할 수 있어야 한다.

즉, 이 명세서의 핵심은 **시나리오 중심 상태머신(state machine)을 잘 설계하는 것**이다.

# 2. 데모 범위와 전제

## 2.1 데모가 시연하는 핵심 흐름

WillChain.pdf 14페이지의 흐름과 사용자가 수정한 시나리오 바를 기준으로, 본 데모는 아래 22개 stage를 순서대로 시연한다.

1. WillChain 가입
2. 유언 작성
3. 자산 등록
4. 스마트 컨트랙트 작성
5. 유언 등록 이벤트 블록 기록
6. 유언 수정
7. 자산 재등록
8. 스마트 컨트랙트 수정
9. 유언 수정 이벤트 블록 기록
10. 재산 토큰화
11. 재산 토큰화 이벤트 블록 기록
12. 생전 증여 집행
13. 증여 집행 이벤트 블록 기록
14. 양도인 사망
15. 사망 확인 이벤트 블록 기록
16. 사후 절차 개시 이벤트 블록 기록
17. 유언 열람 승인 이벤트 블록 기록
18. 유언 자동 열람
19. 보험금 지급
20. 보험금 지급 이벤트 블록 기록
21. 상속 집행
22. 상속 집행 이벤트 블록 기록

중요한 점은, **사후 절차 개시 이후 유언 열람은 사용자의 별도 수동 입력 없이 자동으로 이루어진다**는 것이다. 따라서 유언 열람 승인 이벤트 블록은 "열람을 허용하는 온체인 권한 부여 기록"이고, 그 다음 현실 단계의 `유언 열람`은 사용자가 버튼을 눌러 승인하는 단계가 아니라 **자동 공개 결과를 확인하는 안내 단계**로 구현한다.

## 2.2 데모용 가정

PDF는 개념 제안서이므로, 데모 구현을 위해 다음 가정을 명시적으로 둔다.

### 가정 A. 단일 사례 기반 시연

본 데모는 한 명의 양도인(피상속인)과 두 명의 양수인(상속인/수증자)을 사용하는 **단일 가족 사례**를 시연한다.

- 양도인: 양도인
- 양수인1: 본인(사용자)
- 양수인2: 타 양수인
- 유언집행자: 유언집행자
- 보험사: 한화생명(데모상 메인 오퍼레이터)
- 은행: KB 국민은행
- 의료기관: 아산병원
- 예탁결제원: KSD
- 국가기관: 대법원 / 국세청 / 가정법원

### 가정 B. 자산 구성

양도인의 자산은 데모에서 다음처럼 구성한다.

- 아파트 1채 (부동산)
- 자동차 1대
- 현금성 예금
- 사망보험금 청구권

### 가정 C. 증여와 상속이 둘 다 등장하는 시나리오

데모에서는 **생전 증여**와 **사후 상속**이 모두 등장해야 한다. 그래야 PDF의 메시지인 “살아있는 지금의 증여부터 죽음 이후의 상속까지”를 한 화면에서 보여줄 수 있다.

따라서:

- 생전에는 일부 토큰화 자산이 증여된다.
- 사후에는 잔여 자산과 보험금이 상속된다.

### 가정 D. 날짜는 데모 continuity를 위해 보완

PDF 14페이지는 모든 날짜가 세세하게 적혀 있지 않거나 발표용 축약일 수 있다. 따라서 본 명세서는 **PDF 순서를 보존하되, 데모에 필요한 날짜를 일관성 있게 보완**한다. 이 날짜는 법적 효력 설명용이 아니라 **시연 흐름 이해용**이다.

# 3. 블록체인 네트워크 참여 기관

좌측 참여 주체 영역에서 표현되는 기관은 아래와 같다.

## 3.1 국가기관

### 대법원
- 사망에 따른 가족관계 등록 및 사법적 판단의 관할 기관
- 본 데모에서는 사망 확인 이벤트 기록 시 가족관계 등록/법적 사망 상태 확인 주체로 등장
- 사후 절차 개시 단계에서 법적 확인 주체로 등장 가능

### 국세청
- 상속세/증여세 부과 및 징수 담당 기관
- 생전 증여 집행, 사후 상속 집행 이벤트 승인 과정에서 세무 확인 주체로 등장
- “증여세 납부 완료 확인”, “상속세 신고/확정 확인”과 같은 승인 문구를 제공

### 가정법원
- 상속 분쟁의 조정 및 재판 관할 기관
- 기본 시나리오에서는 직접 개입하지 않음
- 단, 잠재적 법적 확인 주체로 메타정보에 노출 가능
- 분쟁 없는 Happy Path 데모에서는 비활성(회색) 상태가 주로 유지됨

## 3.2 예탁결제원
- 토큰증권 발행 심사 및 전자등록 주관 기관
- 재산 토큰화 이벤트 승인 시 핵심 참여자
- 부동산/자동차 등 실물자산이 토큰화된 뒤, 토큰 등록이 완료되었음을 승인

## 3.3 은행
- 양도인 유동자산 예치 및 관리
- 생전 증여 시 일부 자산 이전 결과 표현에 관여
- 상속 집행 시 현금성 자산 정산 결과 표현에 관여
- 다만 온체인 승인권자로 항상 등장하는 것은 아니고, 실제 자산 이동의 오프체인 실행 결과 제공자로 표현 가능

## 3.4 보험사
- 양도인 명의 사망보험금 지급 기관
- **블록체인 네트워크 오퍼레이터**
- 거의 모든 핵심 단계에서 개입
- 오프체인 문서 등록, 이벤트 발행, 승인 집계, 사용자 알림 제공, 보험금 지급 실행의 중심 주체

## 3.5 의료기관
- 사망 사실을 의학적으로 확인하고 사망진단서를 발급
- 유언 능력 판단에 대한 의학적 소견 제공 가능
- 본 데모에서는 사망 확인 이벤트 승인권자로 핵심 등장

## 3.6 유언집행자
- 양도인 사후 유언의 내용을 집행/실현하는 법적 주체
- 유언 등록/수정, 사후 절차 개시, 유언 열람 승인, 상속 집행에 깊게 관여
- 데모에서 사용자 관점상 “진행 관리자” 역할로 보일 수 있음

## 3.7 양도인
- 재산에 대한 증여 및 사후 절차 의사를 설정하는 주체
- 생전 단계에서 유언 작성, 수정, 자산 등록, 스마트 컨트랙트 설정, 재산 토큰화, 증여 집행의 주체
- 사망 이후 단계에서는 더 이상 UI 상 직접 액션하지 않음

## 3.8 양수인(들)
- 양도인의 의사/유언 또는 법률에 따라 재산을 승계 받는 대상
- 본 데모에서는 양수인1, 양수인2 두 명을 사용
- 생전 증여 수령 승인, 사후 유언 열람 승인, 보험금 수령 확인, 상속 자산 수취 승인 등에 등장

# 4. WillChain Block Schema

WillChain의 온체인 각 블록은 다음과 같은 필드로 이루어진다.

## 4.1 해시 관련
- `Prev_Block_Hash`
- `Block_Hash`

설명:
- 블록 간 연결성과 시간 순서를 보장한다.
- 데모에서는 실제 암호학 계산 대신 deterministic mock hash를 사용해도 되지만, 문자열 포맷은 실제처럼 보여야 한다.
- 예: `0xAB88F1D0...`

## 4.2 이벤트 관련
- `Event_ID`
- `Event_Timestamp`
- `Event_Type`
- `Event_Data_Hash`

설명:
- 현실 사건이 언제, 어떤 유형으로 기록되었는지 나타낸다.
- `Event_ID`는 시나리오에서 유일해야 한다.
- `Event_Type`은 enum처럼 엄격히 관리한다.

## 4.3 유언 및 실물자산토큰 관련
- `Related_Will_ID`
- `RWA_Tokens_ID`
- `OffChain_Reference`

설명:
- 온체인과 오프체인 문서를 연결한다.
- 모든 블록에 모든 필드가 필요한 것은 아니다.
- 예를 들어 사망 확인 이벤트에는 `RWA_Tokens_ID`가 없을 수 있다.

## 4.4 승인 관련
- `Approving_Parties`
- `Signature_Set`

설명:
- 승인 주체 목록과 전자서명 집합
- 데모에서는 “승인됨/대기/해당없음” 상태를 표현하고, hover 블록에서는 최종 signature set을 보여준다.

## 4.5 데모용 블록 렌더링 원칙

블록 hover 박스에는 아래 순서로 표시한다.

1. 블록 제목 (예: `Estate Transfer Event Block`)
2. 상태 배지 (`Pending Signatures` 또는 `Committed`)
3. 필드 목록
4. 승인 현황 요약

예시:

```txt
Estate Transfer Event Block
Status: Pending Signatures
Prev_Block_Hash: 0xAB88...
Event_ID: EVT-0011
Event_Timestamp: 2024-02-20T14:10:32+09:00
Event_Type: Inheritance_Execution_Event
Event_Data_Hash: 0xf34ec19a...
Related_Will_ID: WILL-20240106-KMJ
RWA_Tokens_ID: [APT-SHARE-001, CAR-SHARE-001]
OffChain_Reference: ENC-REF-TRANSFER-001
Approving_Parties: [양도인, 양수인1, 국세청, 보험사]
Signature_Set: [Sig_Donor, Sig_Assignee1, Pending, Sig_Insurer]
Block_Hash: 0x3FC6...
```

# 5. WillChain Events 상세 정의

아래 이벤트 정의는 PDF 내용을 유지하되, 데모 구현에 필요한 수준으로 세분화한 것이다.

## 5.1 유언 등록 이벤트 (Will Registration Event)

### 의미
양도인이 최초 유언을 작성하고, 오프체인 저장소에 암호화 보관된 유언 문서의 존재/참조/해시를 온체인에 기록한다.

### 현실 단계 선행조건
- 양도인이 WillChain에 가입되어 있음
- 유언 초안이 작성됨
- 유언집행자가 지정됨

### 온체인 기록 목적
- “이 시점에 이런 유언이 존재했다”는 사실 증명
- 이후 수정 이력의 기준 버전(anchor) 역할

### 승인권자
- 양도인 (발행)
- 유언집행자
- 보험사

### UI 질문/설명
- 양도인: “현재 작성한 유언 문서를 WillChain에 등록하시겠습니까?”
- 유언집행자: “유언 등록 요청을 검토했습니다. 집행자로서 해당 문서 등록을 승인하시겠습니까?”
- 보험사: “오프체인 문서 암호화 저장 및 등록 이벤트 발행을 확인합니다.”

## 5.2 유언 수정 이벤트 (Will Amendment Event)

### 의미
기존 유언을 수정하고 새 버전을 오프체인에 저장한 뒤, 수정 사실과 최신 참조를 온체인에 기록한다.

### 선행조건
- 기존 유언 등록 완료

### 중요한 규칙
- 이전 유언 버전은 삭제되지 않음
- 최신 상태 포인터만 현재 유효 버전으로 갱신됨

### 승인권자
- 양도인 (발행)
- 유언집행자
- 보험사

### UI 설명
- “기존 유언은 이력으로 보존되며, 최신 버전이 활성 버전으로 지정됩니다.”

## 5.3 사망 확인 이벤트 (Death Confirmation Event)

### 의미
의료기관과 법적 등록 주체가 양도인의 사망 사실을 확인하고 이를 원장에 기록한다.

### 선행조건
- 실제 사망 발생
- 사망진단서 발급
- 가족관계/법적 등록 확인 가능

### 승인권자
- 의료기관
- 대법원 (발행)

### UI 설명
- 의료기관: “사망진단서 발급 완료”
- 대법원: “법적 사망 상태 및 등록 확인”

## 5.4 사후 절차 개시 이벤트 (Post-Process Initiation Event)

### 의미
사망 확인 이후 유언 집행 및 상속 절차를 공식 개시하는 이벤트이다.

### 선행조건
- 사망 확인 이벤트 완료

### 발행권자
- 보험사

### 승인권자
- 유언집행자
- 대법원

### UI 설명
- 보험사: “사후 절차 개시 이벤트를 발행합니다.”
- 유언집행자: “사후 집행 절차를 개시할 준비가 되었음을 승인하시겠습니까?”
- 대법원: “사후 절차 개시의 법적 진행 가능 상태를 승인하시겠습니까?”

### 비고
- 본 이벤트가 커밋되면 스마트 컨트랙트에 의해 유언 열람 권한 부여 절차가 자동으로 후속 실행된다.

## 5.5 재산 토큰화 이벤트 (RWA Tokenization Event)

### 의미
양도인의 자산을 토큰화하고 해당 토큰 식별자를 프로세스에 등록하는 이벤트이다.

### 선행조건
- 자산 등록 완료
- 스마트 컨트랙트 설정 완료

### 발행권자
- 양도인

### 승인권자
- 예탁결제원
- 보험사

### UI 설명
- 양도인: “등록 자산을 토큰화 프로세스에 올리시겠습니까?”
- 예탁결제원: “전자등록 심사 결과를 승인하시겠습니까?”
- 보험사: “토큰화 결과를 네트워크에 반영하는 것을 승인하시겠습니까?”

## 5.6 유언 열람 승인 이벤트 (Will Access Authorization Event)

### 의미
사후 절차 개시 이후, 스마트 컨트랙트가 유언 열람 가능 상태를 자동 트리거하고 그 권한 부여 사실을 원장에 남기는 이벤트이다.

### 선행조건
- 사후 절차 개시 이벤트 완료
- 유언 최신본이 오프체인에 보관되어 있음

### 발행권자
- 스마트 컨트랙트

### 승인권자
- 유언집행자
- 보험사

### UI 설명
- 유언집행자: “사후 절차 개시에 따라 유언 열람 권한 자동 부여를 승인하시겠습니까?”
- 보험사: “보호 문서 열람 권한 부여 결과를 원장에 반영하는 것을 승인하시겠습니까?”

### 비고
- 본 이벤트는 **사용자 수동 열람 신청 단계가 아니다.**
- 본 이벤트가 커밋되면 다음 현실 stage에서 유언이 자동 열람된 것으로 표시한다.

## 5.7 증여 집행 이벤트 (Gift Execution Event)

### 의미
생전 증여가 실제로 집행되었음을 기록하는 이벤트이다. 토큰화된 재산 또는 오프체인 자산 이전 결과를 원장에 남긴다.

### 선행조건
- 재산 토큰화 이벤트 완료
- 생전 증여 현실 단계에서 자산 이전이 발생함

### 발행권자
- 양도인

### 승인권자
- 양수인(들)
- 국세청
- 보험사

### UI 설명
- 양도인: “생전 증여 집행 결과를 원장에 기록하도록 발행합니다.”
- 양수인: “증여 자산 수령 결과를 승인하시겠습니까?”
- 국세청: “증여세 확인 결과를 승인하시겠습니까?”
- 보험사: “증여 집행 결과를 네트워크 기록으로 확정하는 것을 승인하시겠습니까?”

## 5.8 상속 집행 이벤트 (Inheritance Execution Event)

### 의미
사망 이후 상속 절차에 따라 잔여 재산이 최종 분배되었음을 기록하는 이벤트이다.

### 선행조건
- 사후 절차 개시 이벤트 완료
- 유언 열람 승인 이벤트 완료
- 상속 현실 단계에서 자산 분배가 실제로 수행됨

### 발행권자
- 스마트 컨트랙트

### 승인권자
- 양수인(들)
- 유언집행자
- 국세청
- 보험사

### UI 설명
- 양수인: “상속 자산 수령 결과를 승인하시겠습니까?”
- 유언집행자: “유언 내용에 따른 집행 결과를 승인하시겠습니까?”
- 국세청: “상속세 신고·납부 확인 결과를 승인하시겠습니까?”
- 보험사: “상속 집행 결과를 네트워크 기록으로 확정하는 것을 승인하시겠습니까?”

## 5.9 보험금 지급 이벤트 (Insurance Payout Event)

### 의미
사망보험금의 지급 완료를 기록하는 이벤트이다.

### 선행조건
- 사망 확인 이벤트 완료
- 보험금 현실 지급 단계 완료

### 발행권자
- 보험사

### 승인권자
- 양수인(들)
- 유언집행자

### UI 설명
- 보험사: “보험금 지급 실행 결과를 원장에 발행합니다.”
- 양수인: “보험금 수령 결과를 승인하시겠습니까?”
- 유언집행자: “보험금 지급 절차가 유언 조건과 부합함을 승인하시겠습니까?”

# 7. 상단 시나리오 바 상세 명세

## 7.1 시나리오 바의 목적

상단 바는 “현실 사건”과 “블록체인 기록”이 번갈아 진행된다는 것을 가장 직관적으로 보여주는 핵심 요소이다.

사용자는 상단 바만 보아도 다음을 이해해야 한다.

- 실제 생활/행정/금융 절차가 먼저 일어난다.
- 그 결과가 온체인 이벤트 블록으로 기록된다.
- 어떤 단계는 현실 단계만 있고, 어떤 단계는 현실 후 온체인 기록이 붙는다.
- 사망 전 단계와 사망 후 단계가 구분된다.

## 7.2 형태 규칙

- 2개의 평행한 수평 라인이 존재한다.
  - 위: 현실 이벤트 바
  - 아래: 온체인 블록 바
- 각 바에는 timeline node가 존재한다.
- 현실 이벤트가 발생한 직후, 아래 온체인 바에 대응 블록이 이어진다.
- 대응 관계가 있는 경우 두 바의 노드가 시각적으로 수직 정렬되거나 얇은 연결선으로 연동 표시된다.

## 7.4 현재/완료/미래 상태 표현

각 노드는 아래 3가지 상태를 가진다.

- 완료(completed)
- 현재(current)
- 미래(upcoming)

표현 규칙:

- 완료: 진한 브랜드 색 + 체크 아이콘
- 현재: 확대된 노드 + 펄스 애니메이션

## 7.5 클릭 규칙

- 클릭 시 중앙/좌측/우측/hover 데이터 모두 해당 stage 기준으로 동기화

# 8. 좌측 참여 주체 패널 상세 명세

## 8.1 공통 구조

각 참여 주체 row는 다음 구조를 가진다.

- 왼쪽 원형 로고
- 로고 아래 작은 이름 라벨
- 오른쪽 액션 영역

로고 이미지는 stage 데이터 또는 참여자 프로필 데이터의 `imageSrc` 필드로 관리하며, 예시는 `assets/images/hospital.png`, `assets/images/insurer.png` 와 같다.

예시:

`[원형로고]\n보험사`  |  `오프체인 문서 저장 요청 수신` / `승인` 버튼

## 8.2 로고 상태 규칙

### 활성(active)
- 풀컬러
- 불투명도 100%
- shadow 있음

### 비활성(inactive)
- grayscale
- opacity 35~45%
- hover 없음

### 강조(focus)
- 현재 stage에서 발행권자일 경우
- 컬러 + ring + subtle glow

## 8.3 액션 영역 유형

각 row의 액션 영역은 다음 중 하나를 가진다.

1. 설명 텍스트만 표시
2. `승인` 버튼 표시
3. `발행됨` 버튼 표시
4. `승인됨` 버튼 표시
5. 해당 없음 (빈 영역)

### 승인 버튼 종류
- 진행 가능한 액션 버튼: `승인`
- 이미 승인이 끝난 비활성 버튼: `승인됨`
- 발행권자가 이미 발행한 비활성 버튼: `발행됨`

### 버튼 표기 규칙
- 버튼 문구는 **반드시** `승인`, `승인됨`, `발행됨` 셋만 사용한다.
- `보험금 수령 결과를 승인하시겠습니까?`, `유언 등록 반영을 승인하시겠습니까?` 같은 문구는 버튼 문구가 아니라 **질문/설명 텍스트**로 표현한다.
- 따라서 액션 영역의 설명 문구는 구체적으로 쓰되, 버튼 문구는 통일한다.

예시:
- 설명: `보험금 수령 결과를 승인하시겠습니까?` / 버튼: `승인`
- 설명: `유언 등록 이벤트를 발행했습니다.` / 버튼: `발행됨`
- 설명: `전자등록 심사 승인이 완료되었습니다.` / 버튼: `승인됨`

## 8.4 각 stage에서 가능한 액션 예시

### 유언 등록 이벤트 stage
- 양도인: `유언 등록 요청 내용을 승인하시겠습니까?` + `승인`
- 유언집행자: `유언 등록 검토 결과를 승인하시겠습니까?` + `승인`
- 보험사(발행권자): `유언 등록 이벤트를 발행했습니다.` + `발행됨`

### 유언 수정 이벤트 stage
- 양도인: `유언 수정본 등록 요청을 승인하시겠습니까?` + `승인`
- 유언집행자: `최신 유언 버전 반영을 승인하시겠습니까?` + `승인`
- 보험사(발행권자): `유언 수정 이벤트를 발행했습니다.` + `발행됨`

### 재산 토큰화 이벤트 stage
- 양도인(발행권자): `재산 토큰화 이벤트를 발행했습니다.` + `발행됨`
- 예탁결제원: `전자등록 심사 결과를 승인하시겠습니까?` + `승인`
- 보험사: `토큰화 결과 반영을 승인하시겠습니까?` + `승인`

### 증여 집행 이벤트 stage

# 10. 우측 사용자 UI 패널 상세 명세

우측 패널은 “웹페이지 안의 또 하나의 UI”처럼 보여야 한다.

## 10.1 형태

- 노트북/태블릿 프레임 안에 렌더링
- 내부 스크롤 가능
- 카드 리스트 + 팝업 + 승인 모달 + 타임라인 요약 영역 포함

## 10.2 역할

사용자에게 온체인 raw data를 직접 보여주는 것이 아니라, **사람이 이해하기 쉬운 설명형 인터페이스**를 제공한다.

예시:
- “유언이 등록되었습니다. (24.02.16)”
- “스마트 컨트랙트가 작성되었습니다. 조건 충족 시 자동 집행됩니다.”
- “사망이 확인되었습니다. 사후 절차를 개시하시겠습니까?”

## 10.3 우측 UI의 주요 컴포넌트

1. 진행 현황 헤더
2. 카드형 활동 로그 리스트
3. 현재 stage 관련 팝업/모달
4. 승인 버튼/확인 버튼
5. 요약 정보 배지

각 stage가 완료되면 사용자 UI에 누적되는 카드

카드 필드:

- 상태 배지

# 11. 웹페이지 동작 원리: Stage 기반 상태머신

## 11.1 Stage 정의

웹페이지는 **stage 단위**로 동작한다.

Stage는 다음 두 유형으로 나뉜다.

1. `real_event_stage`
2. `block_commit_stage`

각 stage는 “이 시점에서 화면 전체가 어떤 상태여야 하는가”를 정의한다.

즉 stage 하나만 보면 다음을 모두 알 수 있어야 한다.

- 상단 바에서 현재 위치
- 좌측 참여자별 상태/액션
- 중앙 자산 상태
- 우측 UI의 카드/팝업/버튼
- 하단 이동 가능 여부
- 블록 hover에 보여줄 내용
- stage 완료 시 다음 stage로 어떻게 이동하는지

## 11.2 Stage 전환 원칙

- 사용자가 하단 화살표 클릭 → 이전/다음 stage 이동
- 상단 특정 노드 클릭 → 해당 stage로 즉시 점프
- 현실 이벤트 stage에서 주요 CTA 클릭 → 해당 stage의 완료 상태 반영 후 자동으로 다음 `block_commit_stage`로 이동 가능
- block_commit_stage에서 모든 필요한 승인이 완료되면 해당 블록이 `Committed` 상태가 되고 다음 stage로 이동 가능

## 11.3 블록 stage 내부 진행 방식

모든 block_commit_stage는 내부적으로 3개 소단계를 가질 수 있다.

1. `initiated`
   - 이벤트 발행 주체는 이미 발행 완료 상태
   - 나머지 승인권자는 pending
2. `approving`
   - 사용자가 각 승인 버튼 클릭하며 서명 수집
3. `committed`
   - 모든 필수 서명이 모여 블록 최종 기록 완료

이 소단계는 별도 화면 전환 없이 같은 stage 내부에서 상태 변화로 처리할 수 있다.

# 12. 현실 이벤트 및 블록 기록의 세부 동작 정의

이 섹션은 구현에서 가장 중요하다. 각 stage마다 무엇이 보이고, 어떤 버튼이 있고, 누가 발행 또는 승인하며, 어떤 블록이 만들어지는지 모두 정의한다. 버튼 텍스트는 항상 `승인`, `승인됨`, `발행됨` 셋만 사용한다.

## 12.1 Stage 1 — WillChain 가입 (real_event_stage)
- 상단 현재 위치: `WillChain 가입`
- 활성 주체: 양도인, 보험사
- 좌측 액션: 양도인 `WillChain 가입 요청을 진행 중입니다.`, 보험사 `기본 계정 및 참여자 연결을 준비 중입니다.`
- 중앙 상태: 유언장/자산/스마트 컨트랙트/실물자산토큰은 모두 미등록 상태
- 우측 팝업: `WillChain에 가입하시겠습니까?`
- 완료 액션: `가입하기`
- 완료 후: 가입 완료 카드 추가, Stage 2로 이동

## 12.2 Stage 2 — 유언 작성 (real_event_stage)
- 활성 주체: 양도인, 유언집행자, 보험사
- 좌측 액션: 양도인 `유언 초안 작성 중입니다.`, 유언집행자 `집행자 지정 상태를 검토 중입니다.`, 보험사 `오프체인 저장 준비 중입니다.`
- 중앙 변화: 유언장 `초안 작성됨. 아직 온체인에 등록되지 않음.`
- 우측 팝업: `유언 초안을 작성해주세요.`
- 완료 후: Stage 3이 아니라, 사용자가 수정한 시나리오 바에 맞춰 **자산 등록(Stage 3)** 으로 진행한다.

## 12.3 Stage 3 — 자산 등록 (real_event_stage)
- 활성 주체: 양도인, 보험사, 은행
- 좌측 액션: 양도인 `상속·증여 대상 자산을 입력 중입니다.`, 은행 `예금 정보 확인 중입니다.`, 보험사 `등록 자산 연계 준비 중입니다.`
- 중앙 변화: 자산 목록 `부동산, 자동차, 예금 자산이 등록됨.`
- 우측 팝업: `상속·증여 대상 자산을 등록해주세요.`
- 완료 후: Stage 4

## 12.4 Stage 4 — 스마트 컨트랙트 작성 (real_event_stage)
- 활성 주체: 양도인, 보험사, 유언집행자
- 좌측 액션: 양도인 `생전 증여/사후 상속 조건을 입력 중입니다.`, 유언집행자 `집행 조건을 검토 중입니다.`, 보험사 `자동 실행 규칙 연결 준비 중입니다.`
- 중앙 변화: 스마트 컨트랙트 `자동 집행 조건이 설정됨.`, 자산 목록 `등록 자산이 자동 집행 조건과 연결됨.`
- 우측 팝업: `스마트 컨트랙트 조건을 설정하시겠습니까?`
- 완료 후: Stage 5

## 12.5 Stage 5 — 유언 등록 이벤트 블록 (block_commit_stage)
- 발행권자: 양도인
- 승인권자: 유언집행자, 보험사
- 좌측 액션
  - 양도인: `유언 등록 이벤트를 발행했습니다.` / `발행됨`
  - 유언집행자: `유언 등록 검토 결과를 승인하시겠습니까?` / `승인`
  - 보험사: `유언 등록 반영을 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 유언장 `유언 등록됨. 원문은 오프체인에 암호화 보관 중.`
- 블록 예시 Event_Type: `Will_Registration_Event`

## 12.6 Stage 6 — 유언 수정 (real_event_stage)
- 활성 주체: 양도인, 유언집행자
- 중앙 변화: 유언장 `유언 수정됨. 최신 버전 등록 대기 중.`
- 우측 팝업: `유언 수정 사항을 반영하시겠습니까?`
- 완료 후: Stage 7

## 12.7 Stage 7 — 자산 재등록 (real_event_stage)
- 활성 주체: 양도인, 보험사, 은행
- 좌측 액션: 양도인 `수정된 자산 내역을 다시 등록 중입니다.`, 은행 `변경된 예금/자산 정보를 검토 중입니다.`, 보험사 `수정된 자산 구조를 반영 준비 중입니다.`
- 중앙 변화: 자산 목록 `수정된 자산 목록이 재등록됨.`
- 우측 팝업: `수정된 자산 정보를 다시 등록해주세요.`
- 완료 후: Stage 8

## 12.8 Stage 8 — 스마트 컨트랙트 수정 (real_event_stage)
- 활성 주체: 양도인, 보험사, 유언집행자
- 중앙 변화: 스마트 컨트랙트 `수정된 자산 구조에 맞게 조건이 업데이트됨.`
- 우측 팝업: `스마트 컨트랙트를 수정하시겠습니까?`
- 완료 후: Stage 9

## 12.9 Stage 9 — 유언 수정 이벤트 블록 (block_commit_stage)
- 발행권자: 양도인
- 승인권자: 유언집행자, 보험사
- 좌측 액션
  - 양도인: `유언 수정 이벤트를 발행했습니다.` / `발행됨`
  - 유언집행자: `최신 유언 버전 반영을 승인하시겠습니까?` / `승인`
  - 보험사: `유언 수정 반영을 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 유언장 `최신 유언 버전이 등록됨. 이전 버전은 이력으로 보존됨.`
- 블록 예시 Event_Type: `Will_Amendment_Event`

## 12.10 Stage 10 — 재산 토큰화 (real_event_stage)
- 활성 주체: 양도인, 예탁결제원, 보험사
- 중앙 변화: 실물자산토큰 `토큰 발행 요청됨. 전자등록 대기 중.`, 자산 목록 `일부 자산이 토큰화 준비 상태임.`
- 우측 팝업: `등록 자산을 실물자산토큰으로 전환하시겠습니까?`
- 완료 후: Stage 11

## 12.11 Stage 11 — 재산 토큰화 이벤트 블록 (block_commit_stage)
- 발행권자: 양도인
- 승인권자: 예탁결제원, 보험사
- 좌측 액션
  - 양도인: `재산 토큰화 이벤트를 발행했습니다.` / `발행됨`
  - 예탁결제원: `전자등록 심사 결과를 승인하시겠습니까?` / `승인`
  - 보험사: `토큰화 결과 반영을 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 실물자산토큰 `실물자산토큰 발행 완료.`
- 블록 예시 Event_Type: `RWA_Tokenization_Event`

## 12.12 Stage 12 — 생전 증여 집행 (real_event_stage)
- 활성 주체: 양도인, 양수인 본인, 보험사, 국세청
- 우측 팝업: `생전 증여를 실행하시겠습니까?`
- 본문: `양도인 자산 중 토큰화된 자산의 40%가 양수인 본인에게 지급됩니다. 이후 세무 확인과 원장 기록 절차가 이어집니다.`
- 클릭 직후 변화: 양도인 계좌 감소, 양수인 본인 계좌 증가, 실물자산토큰 일부 이전 애니메이션
- 완료 후: Stage 13

## 12.13 Stage 13 — 증여 집행 이벤트 블록 (block_commit_stage)
- 발행권자: 양도인
- 승인권자: 양수인 본인, 국세청, 보험사
- 좌측 액션
  - 양도인: `증여 집행 이벤트를 발행했습니다.` / `발행됨`
  - 양수인 본인: `증여 자산 수령 결과를 승인하시겠습니까?` / `승인`
  - 국세청: `증여세 확인 결과를 승인하시겠습니까?` / `승인`
  - 보험사: `증여 집행 결과 반영을 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 실물자산토큰 `일부 토큰이 양수인 본인 계좌로 이전됨.`
- 블록 예시 Event_Type: `Gift_Execution_Event`

## 12.14 Stage 14 — 양도인 사망 (real_event_stage)
- 활성 주체: 의료기관, 대법원, 보험사
- 중앙 변화: 보험금 상태 `사망보험금 지급 심사 가능 상태.`, 사후 절차 상태 `사후 절차 개시 대기.`, 양도인 계좌 `계좌 동결 및 상속 절차 연계 상태.`
- 우측 팝업: `사망 사실이 확인되었습니다.`
- 완료 후: Stage 15

## 12.15 Stage 15 — 사망 확인 이벤트 블록 (block_commit_stage)
- 발행권자: 대법원
- 승인권자: 의료기관
- 좌측 액션
  - 대법원: `사망 확인 이벤트를 발행했습니다.` / `발행됨`
  - 의료기관: `사망진단서 확인 결과를 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 사후 절차 상태 `사후 절차 개시 대기.`
- 블록 예시 Event_Type: `Death_Confirmation_Event`

## 12.16 Stage 16 — 사후 절차 개시 이벤트 블록 (block_commit_stage)
- 발행권자: 보험사
- 승인권자: 유언집행자, 대법원
- 좌측 액션
  - 보험사: `사후 절차 개시 이벤트를 발행했습니다.` / `발행됨`
  - 유언집행자: `사후 절차 개시를 승인하시겠습니까?` / `승인`
  - 대법원: `법적 개시 가능 상태를 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 사후 절차 상태 `사후 절차 진행 중.`, 스마트 컨트랙트 `상속 절차 조건이 활성화됨.`
- 다음 stage 연결: Stage 17이 자동 활성화됨
- 블록 예시 Event_Type: `Post_Process_Initiation_Event`

## 12.17 Stage 17 — 유언 열람 승인 이벤트 블록 (block_commit_stage)
- 발행권자: 스마트 컨트랙트
- 승인권자: 유언집행자, 보험사
- 좌측 액션

  - 유언집행자: `자동 열람 권한 부여를 승인하시겠습니까?` / `승인`
  - 보험사: `보호 문서 접근 허용 반영을 승인하시겠습니까?` / `승인`
- 커밋 후 중앙 변화: 유언장 `유언 열람 권한이 자동 부여됨. 승인된 주체가 원문 확인 가능.`
- 블록 예시 Event_Type: `Will_Access_Authorization_Event`

## 12.18 Stage 18 — 유언 자동 열람 (real_event_stage)
- 활성 주체: 양수인 본인, 타 양수인, 유언집행자
- 우측 팝업: `유언이 자동으로 열람되었습니다.`
- 본문: `사후 절차 개시 이후 권한 부여가 완료되어 유언 원문이 자동 열람 가능한 상태가 되었습니다.`
- 버튼: `확인`
- 중앙 변화: 유언장 `유언 원문이 자동으로 열람되었음.`
- 완료 후: Stage 19

## 12.19 Stage 19 — 보험금 지급 (real_event_stage)
- 활성 주체: 보험사, 유언집행자, 양수인 본인, 타 양수인

- 클릭 후 변화: 보험금 상태 `보험금 지급됨.`, 양수인 본인 계좌/타 양수인 계좌에 지급 반영
- 완료 후: Stage 20

## 12.20 Stage 20 — 보험금 지급 이벤트 블록 (block_commit_stage)
- 발행권자: 보험사
- 승인권자: 유언집행자, 양수인 본인, 타 양수인
- 좌측 액션
  - 보험사: `보험금 지급 이벤트를 발행했습니다.` / `발행됨`
  - 유언집행자: `지급 절차 결과를 승인하시겠습니까?` / `승인`
  - 양수인 본인: `보험금 수령 결과를 승인하시겠습니까?` / `승인`
  - 타 양수인: `보험금 수령 결과를 승인하시겠습니까?` / `승인`

- 블록 예시 Event_Type: `Insurance_Payout_Event`

## 12.21 Stage 21 — 상속 집행 (real_event_stage)
- 활성 주체: 유언집행자, 양수인 본인, 타 양수인, 보험사, 국세청, 은행
- 우측 팝업: `상속 집행을 완료하시겠습니까?`
- 본문: `잔여 예금, 부동산 지분, 자동차 권리가 유언 내용과 승인 결과에 따라 최종 분배됩니다.`
- 클릭 후 변화: 자산 목록 `잔여 자산 분배 완료.`, 실물자산토큰 `잔여 토큰 분배 완료.`, 양수인 본인 계좌/타 양수인 계좌 `최종 상속 자산 배분 완료.`, 사후 절차 상태 `사후 절차 완료.`
- 완료 후: Stage 22

## 12.22 Stage 22 — 상속 집행 이벤트 블록 (block_commit_stage)
- 발행권자: 스마트 컨트랙트
- 승인권자: 양수인 본인, 타 양수인, 유언집행자, 국세청, 보험사
- 좌측 액션
  - 시스템 배지 또는 스마트 컨트랙트 row: `상속 집행 이벤트가 자동 발행되었습니다.` / `발행됨`
  - 양수인 본인: `상속 자산 수령 결과를 승인하시겠습니까?` / `승인`
  - 타 양수인: `상속 자산 수령 결과를 승인하시겠습니까?` / `승인`
  - 유언집행자: `유언 내용에 따른 상속 집행 결과를 승인하시겠습니까?` / `승인`
  - 국세청: `상속세 확인 결과를 승인하시겠습니까?` / `승인`
  - 보험사: `상속 집행 결과 반영을 승인하시겠습니까?` / `승인`

- 블록 예시 Event_Type: `Inheritance_Execution_Event`

# 13. 블록 Hover 동작 명세

## 13.1 Hover 트리거
- 온체인 블록 바 노드 hover 또는 mousedown 시 표시

## 13.2 Hover 박스 내용
- 블록 제목
- 이벤트 타입 설명
- 필드 목록
- 현재 승인 상태
- 관련 오프체인 참조
- 관련 자산 토큰 ID

- `Status: Pending Signatures`
- `Signature_Set`에 Pending 포함

### 완료 후
- `Status: Committed`
- 모든 서명 기입

# 16. 스테이지별 설정값 작성 원칙

각 stage 데이터는 아래 원칙으로 작성해야 한다.

## 16.1 immutable snapshot 방식

각 stage는 이전 stage에서 diff만 저장하지 말고, **해당 시점의 전체 화면 상태를 완성형 snapshot**으로 들고 있는 것을 권장한다.

이유:
- 특정 stage 클릭 시 바로 렌더링 가능
- 디버깅 쉬움
- 발표용 데모에서 안정적

## 16.2 카드 누적 방식

`userLogCards`는 현재까지 누적된 카드 목록 전체를 stage마다 담는 것이 좋다.

## 16.3 block stage의 approval 상태

block stage는 사용자 클릭에 따라 내부적으로 signature set이 바뀔 수 있으므로:
- 초기 정의 데이터는 pending 상태
- 런타임 state에서 승인 결과를 별도로 관리

권장:
- `stageDefinitions`는 초기 상태만 제공
- `useDemoStageStore` 같은 곳에서 승인 여부 상태를 runtime으로 관리

# 20. 최종 요약

이 데모 웹페이지는 단순히 PDF 내용을 예쁘게 재현하는 수준이 아니라, **WillChain 시나리오를 현실 세계 사건과 블록체인 기록으로 나누어 단계적으로 체험하게 만드는 인터랙티브 설명 시스템**이어야 한다.

가장 중요한 구현 포인트는 다음 4가지다.

1. **상단 시나리오 바가 전체 흐름을 정확히 보여줄 것**
2. **좌측 참여 주체 패널이 각 단계의 승인 구조를 명확히 보여줄 것**
3. **중앙 상태 패널이 유언/자산/토큰/계좌 상태 변화를 직관적으로 보여줄 것**
4. **우측 사용자 UI가 일반 사용자도 이해 가능한 언어로 절차를 설명할 것**

그리고 이 모든 것은 결국 **stage 데이터 파일 하나를 중심으로 제어 가능해야 한다.**

즉 구현자는 이 문서를 바탕으로 `specification.md`를 최상위 기준으로 삼고, 이후 `scenario.ts` 혹은 `stages.ts` 파일에 위 stage 구조를 그대로 데이터화하여 구현하면 된다.
